hearingSlots endpoint — multi-day Crown timeout

Endpoint: GET /listing-query-api/.../hearingSlots Symptom: request times out Branch: team/ccsph2n
TL;DR. The multi-day Crown path forces an unpaginated pageSize=10000 fetch over an extended date window, runs the same heavy SQL twice (once to count, once to page), groups by 25 columns with a LEFT JOIN allocated_listings, has no index matching the actual predicate set, and logs the full entity list at INFO. Several O(N) costs stack — alone any one would be tolerable, together they exceed the request budget.

3 · Findings, in order of impact

#1

Multi-day path discards pageSize=10 and refetches with pageSize=10000 on an extended window

High

SlotsSearchService.java · lines ~89–110

For your request, daysNeeded = 720 / 360 = 2 and weekendBuffer = 2 * ((2/5) + 1) = 2, so the discovery window becomes 31 + (2 − 1) + 2 = 34 days. The repository is then called with UNPAGINATED_PAGE_SIZE = "10000", fetching every matching row in that window before any pagination is applied.

Why it matters: across all rooms in oucodeL2Code=1 with panels ADULT+YOUTH and AD sessions over 34 days, this is typically thousands of rows.

#2

The repository runs the same heavy SQL twice just to know totalCount

High

CourtScheduleRepository.getCourtSchedules · lines 990–1014

String countQuery = buildFullQuery(requestParam).replace(GET_HEARING_SLOTS_QUERY_PAGINATION, "");
List<CourtSchedule> allSchedules = executeQuery(countQuery, queryParamsForCount);
int totalCount = allSchedules.size();      // entire result set materialised just to .size() it

String paginatedQuery = buildFullQuery(requestParam);
List<CourtSchedule> paginatedSchedules = executeQuery(paginatedQuery, queryParamsForResult);

The "count" branch isn't a SELECT COUNT(*) — it pulls every column for every row, then takes .size() of the list. Paired with #1, the two queries do near-identical work twice per request.

#3

The base query groups by 25 columns and aggregates over a LEFT JOIN allocated_listings

High

CourtScheduleRepository.GET_HEARING_SLOTS_QUERY_GROUP_BY · line ~325

The SELECT computes three SUM(CASE WHEN ...) aggregates (totalbookedformorning, totalbookedforafternoon, totalbooked) against the joined allocated_listings, then collapses through a 25-column GROUP BY on cs.id, cs.court_listing_profile_id, cs.oucode, cs.court_room_id, cs.court_room_number, cs.court_house_id, cs.court_house_name, cs.court_room_name, cs.operational_unit, cs.rota_business_type, cs.panel, cs.court_session, cs.active, cs.is_slot_based, cs.session_start, cs.max_slot, cs.max_duration_mins, cs.available_slot, cs.available_duration_mins, cs.support_ad_split, cs.max_ad_morning_duration, cs.max_ad_afternoon_duration, cs.is_overbooking_allowed, cs.is_draft, cs.jurisdiction. The wide GROUP BY plus join expansion is where Postgres spends most of its time on this query.

#4

No index covers the actual predicate set

High

liquibase changesets 006, 010, 014, 036, 051

IndexColumnsHelps this query?
compo_panel_idx(panel, session_start, oucode, rota_business_type, court_session, court_house_name, court_room_name, available_duration_mins, active)Partially — leading panel matches IN ('ADULT','YOUTH') but the planner often skips this with multi-value IN.
court_schedule_oucode_session_start_idx(oucode, session_start)No — this query filters by operational_unit, not oucode.
court_act_business_date_session_idx_am/_pm(oucode, court_room_id, rota_business_type, session_start, court_session, is_draft) partial WHERE activeNo — same oucode mismatch, and no court_room_id filter in this query.

Combined predicate the planner sees: active & operational_unit & panel IN (...) & session_start BETWEEN ... & is_draft & jurisdiction & court_session IN (...) & is_slot_based. No index supports this shape, so Postgres falls back to a sequential scan of court_schedule.

#5

INFO-level logging serialises the full entity list each request

Med

CourtScheduleRepository.getCourtSchedules · lines 997, 1004

LOGGER.info("GET_HEARING_SLOTS_QUERY_MANDATORY_PARAMS ******* allSchedules : {}", allSchedules);
...
LOGGER.info("GET_HEARING_SLOTS_QUERY_MANDATORY_PARAMS ******* paginatedSchedules : {}", paginatedSchedules);

SLF4J's {} is lazy-evaluated but still calls toString() on every element when INFO is enabled. With 10,000 entities of ~30 fields, that is a measurable per-request cost on top of the DB time.

#6

filterForMultidayAvailability is O(rooms × days × daysNeeded) in Java

Med

SlotsSearchService.java · lines 139–170

After the DB hit, schedules are grouped by courtRoomId + businessType + ouCode, each group is sorted, and for each schedule the code walks daysNeeded forward calling isValidMultidayStart. With many rooms over a 34-day window the post-processing isn't trivial — but it's dwarfed by the DB cost in items #1–#4.

#7

The look-ahead window is queried, aggregated, then thrown away

Low

SlotsSearchService.getMultidayCourtSchedules · line ~123

multidayResults.removeIf(cs -> cs.getSessionDate().isAfter(originalEndDate));

The weekend-buffer days exist only to validate the last few candidate starts. They are pulled, joined, aggregated, deduplicated, grouped, then discarded.

4 · Recommended fixes, ranked by ROI

# Fix Effort Impact Notes
1 Eliminate the double DB hit Low High Use COUNT(*) OVER () as an extra column in GET_HEARING_SLOTS_QUERY_GROUP_BY and read it from the first row. Drop the separate count query and the .size()-on-full-list pattern. Pure repository change; no API change.
2 Demote INFO logs / stop logging entity lists Trivial Med The three LOGGER.info(... : {}, allSchedules/paginatedSchedules) lines in getCourtSchedules should log .size() only, or move to DEBUG.
3 Add a composite index that matches the real predicate shape Low High Candidate: (operational_unit, panel, jurisdiction, session_start, is_draft, court_session) WHERE active = true. Validate with EXPLAIN ANALYZE against ste data. Single liquibase changeset; use CREATE INDEX CONCURRENTLY in prod-like envs.
4 Slim the multi-day "discovery" query to a lightweight projection Med High When isMultidayCrownSearch, run a thin SELECT (no LEFT JOIN allocated_listings, no SUM(CASE WHEN), narrow column set) just to find which rooms have daysNeeded consecutive sessions. Re-hydrate only the surviving rows via getCourtSchedulesByIdList. The expensive aggregation then runs on ~10 rows instead of ~10k.
5 Push consecutive-availability into SQL with a window function High High Use the session_start - row_number() OVER (PARTITION BY court_room_id ORDER BY session_start) "consecutive island" trick to find daysNeeded-blocks inside the DB, then LIMIT pageSize OFFSET .... Eliminates the unpaginated fetch and the Java group/walk entirely. Worth attempting only after #1–#4.
6 Cap the date window at the validator Trivial Med Reject Crown multi-day requests where sessionEndDate - sessionStartDate > 31 (or 14) days. Defensive only; doesn't fix the underlying cost, but stops runaway queries from scripted callers and UI bugs.
7 Drop the look-ahead-throwaway pattern Low Low Query only the original window; short-circuit isValidMultidayStart when the remaining days run past originalEndDate. Pairs naturally with #4.
Why "two queries, no LIMIT" is the anti-pattern to fix first. It's a classic shape when pagination is bolted on after the fact — the count branch grew into a full row fetch because someone needed totalCount and didn't have the appetite to refactor the SQL. The fix is small and self-contained; it halves DB load even before any index work.

5 · Suggested sequencing

  1. Quick wins. Fix #1 and #2 together in one PR. Roughly 20 lines, instantly halves DB cost and removes log overhead. Safe to ship by itself.
  2. Index. Fix #3 in a single liquibase changeset. Coordinate the EXPLAIN ANALYZE on a ste-equivalent dataset; in any live environment use CREATE INDEX CONCURRENTLY.
  3. Architectural fix. Fix #4 — the slim discovery query is where the multi-day path really wants to land. Own PR with new repository method, integration tests for the discovery + re-hydrate pattern.
  4. Only if still over SLA. Move on to #5, #6, #7 — each is independently safe but worth measuring before adopting.
